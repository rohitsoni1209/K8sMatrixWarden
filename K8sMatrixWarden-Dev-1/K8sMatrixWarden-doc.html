Overview

K8sMatrixWarden is an AI-powered, Kubernetes-native security platform that continuously scans and monitors clusters. A multi-agent architecture — Orchestrator, Scanner, Runtime — collaborates to detect vulnerabilities, map attack surfaces, correlate static findings with live runtime behaviour, and enforce compliance, all reachable from a CLI, a conversational chat, or any MCP-connected AI agent. It is detect-and-report only — read-only on every surface, with no apply/remediate path.

Every check is a Rule — a technique-level detection that self-declares which MITRE tactic, OWASP category, CIS control, and NSA/CISA guidance it maps to. That means the exact same engine answers "scan the production namespace", "scan for Privilege Escalation", and "scan only Container Escape" — the request just changes which rules get selected, never the code path that runs them.

Architecture

Three agents, one shared registry, one execution path for every kind of request — resource-scoped, tactic-scoped, or plain English — all read-only.

AgentPurposeRuns as
OrchestratorClassifies intent, extracts scope, resolves the selector against the registry, manages confirmationsSingle entry point
ScannerRegistry-driven scanning across 10 domain shards; resolves any scan to a rule setDeployment (HPA)
RuntimeReal-time syscall + K8s audit-event matching against the same tagged-rule pattern; correlates live events to static findings + detects driftDaemonSet, 1/node
◈

The Orchestrator never scans directly — it compiles every request, from any entry point (CLI flags, chat, or an MCP tool call), into one internal ScanRequest object and delegates. That's what makes "scan a namespace" and "scan for Persistence" the same code path.

The Core Idea — Two Orthogonal Axes

The Scanner is built on one deliberate separation everything else depends on:

  • Domain = the execution boundary. Rules are grouped by the evidence pattern they share — pods, the RBAC graph, image layers, network objects, cloud IAM. This is where code, the evidence fetch, and scoped RBAC live.
  • MITRE tactic / framework = cross-cutting tags. A rule carries tags owned by no single shard. A hostPath mount legitimately serves Persistence and Privilege Escalation and Lateral Movement — it lives in one shard but is reachable from three different tactic queries.

The Rule Model

The atomic unit of detection self-declares its own taxonomy — authored once, indexed centrally at startup, so "scan for Persistence" is an O(1) lookup, never a full-registry scan.

rule anatomy
Rule(
  "workload-privileged-container", "Privileged container",
  owning_shard="workload_pod_security",
  resource_scope=["Pod"], severity=CRITICAL,
  detection_method=STATIC_CONFIG, check=_priv,
  mitre=[MitreTag(PRIVILEGE_ESCALATION, "T1610", "Deploy Container"),
         MitreTag(EXECUTION,            "T1610", "Deploy Container")],
  owasp="K01", cis=["5.2.2"], nsa_cisa=["Pod Security"],
)

At startup, every shard registers its rules and the MITRE Mapping Engine indexes every tag into six lookup tables: tactic, technique, OWASP, CIS, NSA/CISA, and module. A selector resolves against these indexes directly — never a scan across all 56 rules.

✓

Anti-drift by construction. Tags live with the rule's logic, not in a separate mapping file that can go stale. doctor fails the build if any rule references a MITRE technique id absent from the vendored ATT&CK-for-Containers taxonomy, or if a composite alias targets a rule id that doesn't exist.

Detect-and-Report Only

K8sMatrixWarden never mutates the cluster. Every surface — CLI, chat, MCP, and the web dashboard — is read-only: scanning uses get/list/watch only, and every output (reports, threat matrix, attack path, dashboard, MCP responses) is derived from that single read-only snapshot.

No write / apply path anywhere

There is deliberately no remediation or apply tool on any surface. Findings still carry everything needed to fix them by hand — a Summary, Standards & Benchmark mapping, the MITRE ATT&CK mapping, real-world Impact, and step-by-step Validation commands to reproduce and verify — but the tool itself stops at detect and report, so it can run against production with a least-privilege read-only identity and no possibility of an accidental change.

◈

The read-only invariant is enforced by a standing test, tests/test_mcp.py::test_no_remediation_or_apply_tool_is_exposed, which fails the build if a write-capable tool name ever appears on the MCP surface. The generated RBAC (k8smatrixwarden roles) is get/list/watch only for the same reason.

Every runtime detection is tagged surface=runtime — the explicit split from the Scanner's point-in-time surface=scan rules. Scan × Runtime correlation — the differentiator

Scanning alone is table stakes (Trivy, kubescape, and kube-bench all do it). The differentiator here is correlation + causality: not "here's a weakness" but "this weakness is being exploited right now, and here's the kill-chain." A static finding says "this config is weak" (point-in-time); a runtime alert says "this behaviour just happened" (live stream). Joined on the MITRE tactic they already share, they answer the question a responder actually has.

SignalWhat it meansMCP tool
confirmedRuntime alert matches a static finding on the same tactic + same namespace — the weakness is being acted on.correlate_runtime
corroboratedSame tactic — behaviour aligns with a known static gap.
runtime-onlyLive behaviour with no matching static finding — something the scan never predicted (often the most interesting).
driftRuntime behaviour that contradicts a Pod's declared posture (uid 0 despite runAsNonRoot, a write despite readOnlyRootFilesystem, a privileged-only op from a non-privileged pod) — the strongest signal, because a control the operator thinks is enforced is not.detect_drift

Both are pure functions of (findings, events) — no cluster access, no re-scan. Events that can't be attributed to a scanned pod are skipped, not guessed, keeping false positives near zero. From the hit cells of the threat matrix, build_attack_path chains a full kill-chain (Initial Access → … → Impact) with entry points and a reaches-Impact flag; intelligent_scan does the whole thing in one call — parse the request, detect the cluster, scan, build the matrix, and derive the attack path.

Kubernetes Threat Matrix

Every scan projects its findings onto the 9-tactic Kubernetes Threat Matrix (MITRE ATT&CK for Containers). Each technique cell is painted one of three states — so the matrix reads as an honest coverage map, not just a list of hits:

hit a finding fired (coloured by worst severity)  ·  covered a rule exists, nothing found this scan  ·  gap no detection rule yet

bash
k8smatrixwarden matrix --mock                    # the matrix for a fresh scan

It ships in the markdown/JSON/HTML reports, as a heatmap on the web dashboard, and over MCP via build_threat_matrix — one matrix, every surface.

Web Dashboard

A zero-dependency Security Dashboard: a stdlib http.server back end serving a vanilla-JS single-page app — no framework, no build step, no CDN. The shell fetches GET /api/dashboard once and renders everything client-side across seven tabs.

A report selector in the header switches the whole dashboard to any saved scan (GET /api/dashboard?scan_id=…), and every finding is click-through, opening straight to its card in the full report (/report/<scan_id>#<finding>). The interface uses a professional, low-emoji visual style, and all timestamps are shown in IST (e.g. 19 Jul 2026, 01:13 IST).

bash
# start it (binds 127.0.0.1:8080 by default)
k8smatrixwarden web

k8smatrixwarden web --no-scan                 # serve saved reports read-only
k8smatrixwarden web --reports-dir ./reports   # override the shared report store
One shared history across CLI, MCP, and web. By default all three surfaces read/write the same store — ~/.k8smatrixwarden/reports (override for all at once with $K8SMATRIXWARDEN_REPORTS_DIR) — resolved independently of the working directory, and every scan is persisted automatically. So a scan you run from the CLI, or that an LLM runs through the run_scan/intelligent_scan MCP tools, shows up in this dashboard's Scan history with no extra step. Name a scan (CLI --name, MCP scan_name, or the dashboard's Scan name field) and its report is titled <name> + date + time.

Security: binding beyond localhost

The dashboard binds 127.0.0.1 by default and has no authentication. Two things change the moment it binds a routable address:

  1. anyone who can reach the port can read every saved report, and start scans unless --no-scan is set;
  2. a kubeconfig in the request body is arbitrary code — loading one executes its credential plugin (aws eks get-token, gke-gcloud-auth-plugin, kubelogin) as the user running the server. That is how cloud auth works and the kubernetes client does the same; on loopback the only caller is the operator, but on a routable bind it is remote code execution.

So on a non-loopback bind the server refuses a request-supplied kubeconfig — both kubeconfig (a path) and kubeconfig_content (an upload), since either names a config whose plugin would run here. POST /api/scan returns 403, and the Scan tab shows the explanation instead of a control that would always fail. Mock scans and live scans against the server’s own kubeconfig/context keep working, and startup prints a warning naming the active mode.

CommandResult
k8smatrixwarden webDefault. Loopback; the kubeconfig path box and file picker work.
k8smatrixwarden scan --live --kubeconfig …Preferred for a remote host — scan from the CLI on the server itself.

Light & dark mode, full-width layout

Every page — the dashboard, the coverage matrix, and each per-scan HTML report — carries a theme button in the top bar. It flips between light and dark, and the choice is remembered in localStorage across pages and sessions; with nothing chosen the page follows the OS prefers-color-scheme. Every colour comes from a CSS variable, so one attribute flip re-themes the whole surface — heatmap, tables, charts — with no reload. The layout uses the full browser width (up to 1720px) instead of a narrow centred column, so the 9-column threat matrix, the findings table and the attack map get real room; narrow viewports still collapse cleanly.

Scan health — an unread cluster is never shown as a clean one

A scan that could not read the cluster produces zero findings because nothing was inspected, which would otherwise score as Excellent. The dashboard refuses to render that:

  • the rating becomes Unknown and the risk / high+ KPIs show —, not 0.0;
  • a red “Scan incomplete — this cluster was not read” banner sits above the tabs (visible on every one) and lists each unreadable resource type;
  • Overview, Findings, Threat Matrix, Attack Path and Attack Map each repeat a short “Not a result” note, so no single view can be screenshotted out of context;
  • the saved report says the same in every format (HTML, markdown, text, terminal, PDF, JSON), and POST /api/scan returns evidence_ok + warnings[].

A partially readable cluster keeps its real rating but gets an amber “Partial coverage” banner naming the missing types — those findings are real, just incomplete.

The tool detects and reports only — there is no remediation/apply path on any surface. Saved-report lookups are path-traversal-guarded, and --no-scan serves existing reports fully read-only.

MCP Server

30 tools across four layers, so any MCP-compatible AI agent — Claude, Cursor, VS Code, Windsurf — can do everything the CLI can: scan, audit, project the threat matrix and kill-chain, correlate a live runtime feed against static findings, and export reports. The whole surface is read-only — there is no apply/remediate tool. Every tool's docstring is its LLM-facing description, and every parameter carries its own schema description (via Annotated[T, Field(description=…)]) — a fully documented argument list for all 150 parameters. Scans over MCP save to the shared store by default, so an LLM-driven scan appears in the web dashboard's Scan history.

.cursor/mcp.json
{
  "mcpServers": {
    "k8smatrixwarden": {
      "command": "python",
      "args": ["-m", "k8smatrixwarden", "mcp"]
    }
  }
}

k8smatrixwarden mcp --list-tools lists all 30 without starting the server; k8smatrixwarden mcp starts it over stdio (requires pip install -e ".[mcp]").

⛔

Glossary

Domain ShardA scanner plugin owning rules that share a data-source/evidence pattern; the execution boundary.
RuleThe atomic technique-level check; self-declares its MITRE/OWASP/CIS/NSA tags.
Rule RegistryCatalog of every rule; resolves a selector → rule_id set.
MITRE Mapping EngineStartup-built index over rule tags — O(1) tactic/technique/framework/alias lookups.
Composite AliasA named outcome selector (e.g. "Container Escape") expanding to a curated rule set.
Evidence CollectorShared, scope-aware fetch/cache — each resource kind fetched once per scan. Live mode preflights connectivity and authentication: an unreachable cluster or a 401 fails fast with one clear message (for a kubeconfig exec plugin it re-runs the plugin itself and quotes its real error — provider-agnostic, so EKS/GKE/AKS each surface their own, and the legacy auth-provider mechanism is named too), while a resource type it merely can't read (RBAC 403 / absent API group) is skipped-with-warning rather than aborting the scan. If nothing could be read, the result is flagged evidence_ok:false and rated Unknown — never Excellent.
Attack-Path BonusA risk-score multiplier for a finding that enables multiple tactics along a chain.
Finding anchorA stable, URL-safe slug for a finding; the report gives each card that id so any surface can deep-link to /report/<scan_id>#<anchor>.
MCPModel Context Protocol — the open standard this tool's 30-tool server implements.
Detection surfacescan (point-in-time config analysis, Scanner Agent) vs runtime (live event/behaviour observation, Runtime Agent) — every detection is tagged one or the other.
Threat MatrixThe 9-tactic Kubernetes Threat Matrix; a scan's findings project onto it as a covered/hit/gap heatmap.

Roadmap

K8sMatrixWarden ships today as a complete detect-and-report platform. The following enhancements are planned, each building on infrastructure that is already in place.

  • Native live collectors. Direct in-cluster collectors for ComponentConfig and CloudIAM evidence, complementing today's kube-bench JSON ingestion and cloud IAM inputs — the rule logic is already written and runs identically in mock and live mode.
  • First-class Trivy / kubescape adapters. Wiring the external-tool adapter seams into live subprocess calls, on top of the fully built native CIS engine.
  • In-cluster Falco feed. The correlation and drift layer, Falco-event ingestion (POST /api/runtime, normalize_events), and the deploy_falco helm automation are built and tested; the next step runs Falco in-cluster so falcosidekick streams live events end to end.
  • Resource-level correlation. correlate_runtime joins on MITRE tactic + namespace today; adding k8s.pod.name enrichment from the live Falco feed will tighten confirmed matches to the exact resource.
  • Per-finding causal attack graph. Extending the kill-chain attack path from tactic-ordered chaining into a per-finding causal graph (e.g. pod-A's ServiceAccount reaching binding-B), surfaced across every report format.
  • LLM-assisted intent parsing. Layering an LLM onto the documented Orchestrator.interpret() seam, complementing today's fast deterministic keyword/fuzzy matching.
