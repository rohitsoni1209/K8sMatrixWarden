Overview

K8sMatrixWarden is an MCP-native Kubernetes security platform. Its whole surface — scanning, the CIS benchmark, the threat matrix, kill-chain attack paths, and scan×runtime correlation — is exposed as 30 MCP tools, so an AI agent (Claude, Cursor, VS Code, Windsurf) drives it conversationally. The CLI and web dashboard run the same engine for anyone not on an MCP client. It is detect-and-report only — read-only on every surface, no apply/remediate path.

Every check is a Rule — a technique-level detection that self-declares which MITRE tactic, OWASP category, CIS control, and NSA/CISA guidance it maps to. That means the exact same engine answers "scan the production namespace", "scan for Privilege Escalation", and "scan only Container Escape" — the request just changes which rules get selected, never the code path that runs them.

Running this from an AI agent? That is the intended path — skip to Connect an AI Agent (MCP). This section covers the standalone CLI, which runs the identical engine.

The core engine has zero required dependencies — it runs on the Python standard library alone, against a bundled, deliberately-insecure mock cluster.

Python 3.10 – 3.14, and nothing else. Because the core imports only the standard library, no third-party package can lag a new Python release. Architecture

Three agents, one shared registry, one execution path for every kind of request — resource-scoped, tactic-scoped, or plain English — all read-only.

AgentPurposeRuns as
OrchestratorClassifies intent, extracts scope, resolves the selector against the registry, manages confirmationsSingle entry point
ScannerRegistry-driven scanning across 11 domain shards; resolves any scan to a rule setDeployment (HPA)
RuntimeReal-time syscall + K8s audit-event matching against the same tagged-rule pattern; correlates live events to static findings + detects driftDaemonSet, 1/node
◈

The Orchestrator never scans directly — it compiles every request, from any entry point (CLI flags, chat, or an MCP tool call), into one internal ScanRequest object and delegates. That's what makes "scan a namespace" and "scan for Persistence" the same code path.

The Core Idea — Two Orthogonal Axes

The Scanner is built on one deliberate separation everything else depends on:

  • Domain = the execution boundary. Rules are grouped by the evidence pattern they share — pods, the RBAC graph, image layers, network objects, cloud IAM. This is where code, the evidence fetch, and scoped RBAC live.
  • MITRE tactic / framework = cross-cutting tags. A rule carries tags owned by no single shard. A hostPath mount legitimately serves Persistence and Privilege Escalation and Lateral Movement — it lives in one shard but is reachable from three different tactic queries.

Design Decisions & Deliberate Deviations

Five choices shape everything else in the codebase, and four places where the implementation knowingly departs from the original architecture specification. Both lists are here so a reader never has to guess whether something is an oversight or a decision.

The five load-bearing decisions

DecisionWhy
Pure-stdlib coreThe engine, rules, CLI, scoring and reporting import nothing outside the standard library, so python -m k8smatrixwarden scan --mock works on a bare Python install with no pip step at all. kubernetes, rich, mcp and fpdf2 are optional extras, imported lazily, each with a graceful fallback.
Rules as code, data as JSONRules are Rule instances constructed in each shard — type-checked, fast, and free of a YAML parser dependency. Only taxonomy, config and fixtures are data files, and they are JSON.
camelCase evidence everywhereLive mode reads raw Kubernetes API JSON (camelCase), matching the mock fixtures exactly, so a rule's field access — spec.securityContext.privileged — is byte-identical in mock and live mode. The client's to_dict() would return snake_case and force every rule to branch per mode; that entire class of drift bug was designed out rather than tested for.
Mock mode is first-classNot a test fixture bolted on afterwards. It is the default when no kubeconfig is reachable, and the bundled cluster is deliberately built to trip a wide spread of rules so the whole pipeline is demonstrable end to end before anyone points it at production.
Tags authored once, on the ruleRules emit partial findings through rule.finding(...) and the engine attaches the owning rule's taxonomy automatically. A tag can never disagree with itself between a rule and its findings, because it is only ever written in one place.

Deliberate deviations from the specification

DeviationJustification
Saturating cluster risk — 10·raw/(raw+K) instead of the spec's literal Σ/max_possible·10max_possible is undefined and unbounded for an arbitrary cluster — there is no ceiling on how many pods a cluster can have. Saturation is monotonic, deterministic, bounded to [0,10], and lets a handful of criticals dominate, which is what the specification's own rating bands describe. See the scoring math.
Rules authored in Python where the spec's examples show YAMLThe target environment has no YAML parser, and Python rules are type-checked and dependency-free. The behaviour is identical to the spec's intent; only the authoring format differs.
Optional, not required, third-party packagesRequiring kubernetes/rich/mcp would make the tool un-runnable out of the box. Every extra degrades gracefully when absent.
The Attack Surface shard analyses across shards rather than re-emitting their findingsRe-emitting would double-count in every severity total and risk score. The value of a cross-cutting shard is the view no single shard has (e.g. ServiceAccount fan-out), not duplication.

None of these alter the architecture. Each is an implementation-level choice made to keep the tool runnable and its numbers honest.

The Risk Score, Exactly

Every number in a report traces back to four multiplications and one normalization. Nothing is weighted by a hidden model.

core/scoring.py
finding_score  = severity_weight × exploitability × blast_radius × path_multiplier
path_multiplier = 1 + 0.25 × (distinct_tactics_on_finding − 1)   # attack-path bonus

raw            = Σ finding_score                       # INFO findings score 0
cluster_risk   = 10 × raw / (raw + K)                  # K = 60.0, saturating
security_score = round((1 − cluster_risk / 10) × 100)  # 0–100
FactorValues
severity_weightCRITICAL 10 · HIGH 7 · MEDIUM 4 · LOW 1 · INFO 0
exploitabilityRemote 3 · Adjacent 2 · Local 1
blast_radiusCluster-wide 3 · Namespace 2 · Pod 1
path_multiplier1.0 for a single-tactic finding, 1.25 for two tactics, 1.5 for three — the attack-path bonus. A writable hostPath mount enables Persistence, Privilege Escalation and Lateral Movement, so it outscores three unrelated single-tactic findings.

INFO findings — including the engine-error findings produced when a rule throws — have weight 0 and cannot move the score. A rule crash therefore degrades coverage visibly without silently inflating or deflating the risk number.

The Rule Model

The atomic unit of detection self-declares its own taxonomy — authored once, indexed centrally at startup, so "scan for Persistence" is an O(1) lookup, never a full-registry scan.

rule anatomy
Rule(
  "workload-privileged-container", "Privileged container",
  owning_shard="workload_pod_security",
  resource_scope=["Pod"], severity=CRITICAL,
  detection_method=STATIC_CONFIG, check=_priv,
  mitre=[MitreTag(PRIVILEGE_ESCALATION, "T1610", "Deploy Container"),
         MitreTag(EXECUTION,            "T1610", "Deploy Container")],
  owasp="K01", cis=["5.2.2"], nsa_cisa=["Pod Security"],
)

At startup, every shard registers its rules and the MITRE Mapping Engine indexes every tag into six lookup tables: tactic, technique, OWASP, CIS, NSA/CISA, and module. A selector resolves against these indexes directly — never a scan across all 60 rules.

✓

Anti-drift by construction. Tags live with the rule's logic, not in a separate mapping file that can go stale. doctor fails the build if any rule references a MITRE technique id absent from the vendored ATT&CK-for-Containers taxonomy, or if a composite alias targets a rule id that doesn't exist.

How Coverage Is Counted

Every technique cell carries one of four states. The distinctions are deliberate — collapsing them would over- or under-state what the tool actually knows.

StateMeaning
hita finding fired here — the cell is painted by its worst finding severity
covereda scan rule exists; nothing was found this scan
runtimeno scan rule, but the Runtime Agent detects it live (Falco / audit / drift)
gapnothing detects this yet — a real coverage gap

Why runtime is not folded into covered

The two answer different questions. A scan rule says “the configuration allows this”; a runtime detection says “this is happening right now”. Several Redguard techniques — a shell spawned inside a container, a crypto-miner starting — are structurally invisible to a configuration snapshot and can only ever be covered at runtime. Reporting them as flat gaps understated the platform’s real coverage; merging them into covered would claim point-in-time coverage the scanner does not have. So coverage_pct (scan only) and coverage_pct_with_runtime are reported as separate numbers, and the Runtime Agent’s catalog is loaded by default in build_threat_matrix so the CLI, reports, dashboard and MCP cannot disagree.

Cells resolve by name, then by ATT&CK id

The Redguard matrix is finer-grained than ATT&CK-for-Containers. Access the K8s API server, Access Kubelet API and Access Kubernetes dashboard are three distinct Discovery techniques that all carry the single id T1613. Matching by id first meant whichever cell held that id absorbed every Discovery-tagged rule, and the other two rendered as coverage gaps — even though rules detecting exactly those techniques were sitting in the registry. The same pattern hid hostPath, host-namespace, sidecar, SSH-in-container and service-account coverage behind the generic ATT&CK name “Deploy Container” (T1610).

Resolution is therefore name before id: naming a Redguard technique verbatim is how a rule opts into that exact cell, while a canonical ATT&CK name (which never matches a Redguard cell name) still resolves by id exactly as before. If you add a rule and want it on a specific cell, name the technique exactly as the matrix names it.

This is the scan-surface counterpart to the Runtime Agent. The Runtime Agent detects log tampering as it happens (Container log cleared/truncated, K8s events deleted — MITRE T1070); this shard detects, at scan time, the posture that makes such tampering unrecoverable. Absence of evidence is not evidence of absence. The three audit-flag rules no-op in that case rather than manufacturing three findings out of missing evidence — the same principle that stops an unread cluster scoring as clean.

Detect-and-Report Only

K8sMatrixWarden never mutates the cluster. Every surface — CLI, chat, MCP, and the web dashboard — is read-only: scanning uses get/list/watch only, and every output (reports, threat matrix, attack path, dashboard, MCP responses) is derived from that single read-only snapshot.

No write / apply path anywhere

There is deliberately no remediation or apply tool on any surface. Findings still carry everything needed to fix them by hand — a Summary, Standards & Benchmark mapping, the MITRE ATT&CK mapping, real-world Impact, and step-by-step Validation commands to reproduce and verify — but the tool itself stops at detect and report, so it can run against production with a least-privilege read-only identity and no possibility of an accidental change.

◈

The read-only invariant is enforced by a standing test, tests/test_mcp.py::test_no_remediation_or_apply_tool_is_exposed, which fails the build if a write-capable tool name ever appears on the MCP surface. The generated RBAC (k8smatrixwarden roles) is get/list/watch only for the same reason.

Scan × Runtime correlation — the differentiator

Scanning alone is table stakes (Trivy, kubescape, and kube-bench all do it). The differentiator here is correlation + causality: not "here's a weakness" but "this weakness is being exploited right now, and here's the kill-chain." A static finding says "this config is weak" (point-in-time); a runtime alert says "this behaviour just happened" (live stream). Joined on the MITRE tactic they already share, they answer the question a responder actually has.

Security: binding beyond localhost

The dashboard binds 127.0.0.1 by default and has no authentication. Two things change the moment it binds a routable address:

  1. anyone who can reach the port can read every saved report, and start scans unless --no-scan is set;
  2. a kubeconfig in the request body is arbitrary code — loading one executes its credential plugin (aws eks get-token, gke-gcloud-auth-plugin, kubelogin) as the user running the server. That is how cloud auth works and the kubernetes client does the same; on loopback the only caller is the operator, but on a routable bind it is remote code execution.

So on a non-loopback bind the server refuses a request-supplied kubeconfig — both kubeconfig (a path) and kubeconfig_content (an upload), since either names a config whose plugin would run here. POST /api/scan returns 403, and the Scan tab shows the explanation instead of a control that would always fail. Mock scans and live scans against the server’s own kubeconfig/context keep working, and startup prints a warning naming the active mode.

CommandResult

k8smatrixwarden scan --live --kubeconfig …Preferred for a remote host — scan from the CLI on the server itself.

Light & dark mode, full-width layout

Every page — the dashboard, the coverage matrix, and each per-scan HTML report — carries a theme button in the top bar. It flips between light and dark, and the choice is remembered in localStorage across pages and sessions; with nothing chosen the page follows the OS prefers-color-scheme. Every colour comes from a CSS variable, so one attribute flip re-themes the whole surface — heatmap, tables, charts — with no reload. Scan health — an unread cluster is never shown as a clean one

A scan that could not read the cluster produces zero findings because nothing was inspected, which would otherwise score as Excellent. The dashboard refuses to render that:

  • the rating becomes Unknown and the risk / high+ KPIs show —, not 0.0;
  • a red “Scan incomplete — this cluster was not read” banner sits above the tabs (visible on every one) and lists each unreadable resource type;
  • Overview, Findings, Threat Matrix, Attack Path and Attack Map each repeat a short “Not a result” note, so no single view can be screenshotted out of context;
  • the saved report says the same in every format (HTML, markdown, text, terminal, PDF, JSON), and POST /api/scan returns evidence_ok + warnings[].

A partially readable cluster keeps its real rating but gets an amber “Partial coverage” banner naming the missing types — those findings are real, just incomplete.

Performance characteristics

  • Evidence is fetched once per scan, cached by kind, shared across every resolved rule. This is the efficiency property the whole architecture exists to guarantee — a tactic scan touching Pods and webhooks fetches each list exactly once, no matter how many rules read them.
  • Rules execute in a thread pool (parallel_rules, default 16). In live mode this overlaps API-bound and CPU-bound rules; each rule is individually guarded, so one throwing an exception becomes an INFO engine-error finding rather than a failed scan.
  • Selector resolution is O(1) index lookups built once at startup — never a scan over rule code.

Known Limitations

Stated plainly, because a security tool that overstates its coverage is worse than one with less of it.

LimitationWhat it means in practice
Synthetic evidence buckets in live modeComponentConfig (control-plane flags) and CloudIAM (workload-identity bindings) have no direct Kubernetes API. Control-plane flags are largely recovered from kube-system static-pod specs; the remainder needs kube-bench JSON, and cloud IAM needs a cloud adapter. The rule logic is already written and fires the moment those buckets are populated.

Natural-language parsing is deterministic, not an LLMCurated synonym patterns plus fuzzy "did you mean" over terms generated from the registry, so phrasing can never drift from what actually exists. An LLM layer would sit on top and resolve against the same index — the seam is Orchestrator.interpret. Over MCP this is moot: the connected model does the interpreting.
Correlation joins on tactic + namespace, not resource nameA same-namespace tactic match is called confirmed; tactic-only is corroborated. Tightening confirmed to require the same resource waits on reliable k8s.pod.name enrichment from Falco.

Remaining threat-matrix gaps

Scan coverage is 77.8%, or 85.2% counting Runtime Agent detections, which are reported as their own state and never folded into scan coverage. The remaining gaps are genuine: CVE scanning (waiting on the Trivy adapter), backdoor-container provenance, cloud service-principal mounts, and three techniques that would require adding a Lateral Movement tactic to existing Credential Access rules. That last one is deliberately not done — extra tactics inflate the attack-path multiplier and would silently move every historical score.

Troubleshooting

ModuleNotFoundError when running python -m k8smatrixwarden

Run it from the repository root (the folder containing the k8smatrixwarden/ package), or install it once with pip install -e . and use the k8smatrixwarden command from anywhere.

Emoji or box-drawing characters render as ? on Windows

The CLI already forces UTF-8 on its own streams. If output is still garbled, run chcp 65001 in the terminal first, or use -o text for a plain-ASCII report.

"Selector matched no rules"

Working as intended — selector terms are validated rather than silently ignored, so a typo fails instead of quietly scanning nothing. Run rules or coverage to see valid values, or scan --dry-run to check resolution before committing to a scan.

--live fails with an import error

pip install -e ".[live]".

"Cannot reach the Kubernetes API server"

The context is valid but the cluster is not answering — stopped, or a wrong endpoint. The tool preflights connectivity and fails fast with the kubectl … cluster-info command to verify, instead of a raw urllib3 traceback. The web surface returns a clean 400 {"error": …}, never a 500.

"Kubernetes API authentication failed" / "the AWS profile could not be found"

The kubeconfig authenticates through a credential plugin and that plugin could not produce a token — nearly always because the cloud profile it names is not configured on this machine. This is not AWS-specific: the tool re-runs whatever exec command the kubeconfig declares, so aws eks get-token, gke-gcloud-auth-plugin and kubelogin each surface their own real error, and the older auth-provider mechanism (gcp/azure/oidc) is handled too. Fix with aws configure --profile <name>, gcloud auth login, or az login.

⛔

Why this is fatal rather than a warning. The Python Kubernetes client only logs an exec-plugin failure and then continues with no credentials at all, so every request returns 401. Treated as a per-resource problem, that made a cluster nobody had read report as 0 findings / Excellent — the worst failure mode a security tool has. A 401 is now fatal at preflight and mid-scan, and such a scan is never saved.

A live scan finished with 0 findings and rating Unknown

The non-fatal form of the same safety net: the cluster answered, but every resource type was refused — typically a blanket 403 from an identity with no read RBAC. Rather than score an empty result as Excellent, the scan is marked evidence_ok: false, rated Unknown, and every surface carries a "Scan incomplete" banner listing what was unreadable. Fix the RBAC and re-scan. Zero findings is only a good result when evidence_ok is true.

Some Forbidden/403 errors during a live scan

This does not abort the scan. Each inaccessible resource type — RBAC-forbidden, or an API group absent on that cluster — is skipped and recorded as a warning, and the scan completes with everything else. The skipped types travel with the saved result (warnings[]), so a re-rendered report still shows the partial-coverage banner. For full coverage, apply the tool's own least-privilege manifest (roles --bind) and scan as that ServiceAccount.

The CIS report shows many NEEDS_NODE controls

Expected on a fresh run — those controls need on-node file reads the Kubernetes API cannot provide. Run kube-bench and pass --kube-bench-json. On a managed cluster also pass --profile eks|gke|aks so the provider-owned control plane is correctly marked NA rather than graded.

How do I confirm the tool itself is not over-privileged?

k8smatrixwarden roles prints the exact minimal ClusterRole generated per shard — only the verbs each shard's rules declared needing, every one of them get/list/watch. roles --bind shows the complete manifest, ServiceAccount bindings included, before you apply anything.

MCP tools do not appear in the client

  • Confirm the server itself is fine: python -m k8smatrixwarden mcp --list-tools should print 30 tool names. If it errors, run pip install -e ".[mcp]".
  • Confirm the client opened the repository root — the shipped configs launch python -m k8smatrixwarden mcp relative to the workspace folder.
  • In VS Code, switch Copilot Chat to Agent mode. Ask and Edit modes cannot call MCP tools.
  • If python is not on your PATH (common on macOS/Linux where only python3 exists), change "command" to python3 in the config file for your client.
  • If you have more than one Python installed, this is almost certainly the cause. The client launches whichever interpreter python resolves to, and if that one lacks the MCP SDK the server dies at import mcp — silently, from the client's point of view. Check which one you are actually getting, then install into that interpreter:
    bash
    python -c "import sys; print(sys.executable)"   # which interpreter is 'python'?
python -c "import mcp"                          # does it have the SDK?

# fix: install into that interpreter, from the repo root
python -m pip install -e ".[mcp]"               # or ".[all]" for every extra
    Read the server's own error before acting on it: since v1.0 it prints the interpreter path, the real exception and the user-site state, which usually identifies the problem outright.
  • "The SDK is installed, my shell proves it, and the client still fails." Then your shell is not the environment the client uses. MCP clients spawn the server with a stripped environment, and a per-user site-packages directory disappears with it — so a package you can import at a prompt becomes invisible to the server. This happens silently whenever pip falls back to a user install, which it does automatically when the interpreter's own site-packages isn't writable (a system-wide Python without admin rights — the common case on Windows).

    The check above gives a false negative here. Reproduce the client's environment instead:

    bash
    # macOS / Linux — disable user-site, as a stripped spawn effectively does
PYTHONNOUSERSITE=1 python -c "from mcp.server.fastmcp import FastMCP"

# Windows PowerShell
$env:PYTHONNOUSERSITE=1; python -c "from mcp.server.fastmcp import FastMCP"

    If that fails while the plain import succeeds, this is your problem. Any one of these fixes it — install into the interpreter's own site-packages (an elevated/admin pip), create a virtualenv and point the client's "command" at its python, or point the client at a different interpreter that already has the SDK in a non-user location.

    Where a package lives is the thing to check — python -c "import mcp, os; print(mcp.__file__)". A path under AppData\Roaming (Windows) or ~/.local (Linux) is a per-user install and is the fragile case.

    An editable install is worth doing regardless: it puts the package on sys.path permanently, so the server starts no matter which working directory the client spawns it in.
  • Restart the client fully after any config or install change — reloading the window is often not enough to re-spawn the server.

Roadmap

K8sMatrixWarden ships today as a complete detect-and-report platform. The following enhancements are planned, each building on infrastructure that is already in place.

  • Native live collectors. Direct in-cluster collectors for ComponentConfig and CloudIAM evidence, complementing today's kube-bench JSON ingestion and cloud IAM inputs — the rule logic is already written and runs identically in mock and live mode.
  • First-class Trivy / kubescape adapters. Wiring the external-tool adapter seams into live subprocess calls, on top of the fully built native CIS engine.
  • In-cluster Falco feed. The correlation and drift layer, Falco-event ingestion (POST /api/runtime, normalize_events), and the deploy_falco helm automation are built and tested; the next step runs Falco in-cluster so falcosidekick streams live events end to end.
  • Resource-level correlation. correlate_runtime joins on MITRE tactic + namespace today; adding k8s.pod.name enrichment from the live Falco feed will tighten confirmed matches to the exact resource.
  • Per-finding causal attack graph. Extending the kill-chain attack path from tactic-ordered chaining into a per-finding causal graph (e.g. pod-A's ServiceAccount reaching binding-B), surfaced across every report format.
  • LLM-assisted intent parsing. Layering an LLM onto the documented Orchestrator.interpret() seam, complementing today's fast deterministic keyword/fuzzy matching.
